## B. False Positives Ruled Out (documented so they aren't re-discovered as "findings" later)

A first-pass grep for the six entity names surfaced several files that do **not** actually use the Prisma models in question — verified by reading each, not assumed from the grep hit alone:

- `apps/web/.../services-configuration-page.tsx`, `update-services-action.ts`, `packages/domain/.../company/services/index.ts`, `get-services-configuration-page.ts` — all matched on the substring `"CompanyService"` inside the *unrelated* identifier `getCompanyServicesConfigurationPage` / `updateCompanyServicesConfiguration` (`CompanyServices`, plural, contains `CompanyService` as a literal substring). None of these files touch the `CompanyService` Prisma model.
- `apps/web/src/site/services/service-groups.ts`'s `CategoryServiceGroup<T>` type name — same kind of substring collision (`CategoryServiceGroup` contains `CategoryService`), unrelated to the `CategoryService` Prisma model.
- `get-requests-list-page.ts` (lines 245, 674–675), `create-request.ts` (line 303) — these are **my own comments** from Phases 13/14, explicitly documenting the *absence* of these dependencies. Confirmed by reading the lines directly.
- `apps/web/src/site/services/public-navigation/coverage.ts` — one comment mentioning "Sector -> Category -> ServiceGroup" as historical lineage/refactor commentary, not a live `Sector` reference.
- `Company.requestMatchingMode` (not one of the six named entities, but tightly coupled to `CompanyService`): the only two repo-wide hits are both comments stating the field is deliberately left untouched. **Zero functional reads or writes remain anywhere.**

## Verdict

**7 verified runtime consumers, not an estimate.** Of these:

- **3 can be removed immediately, with no further design work:** the `resolveInterventionForFunnel` Service/Category traversal, the `CompanyService` write in `update-services-configuration.ts` (the most significant finding — it has had zero readers for multiple phases), and `service-groups.ts` (already fully dead).
- **1 can be partially removed immediately:** `get-request-by-id.ts`'s always-empty `requiredServices` half.
- **3 remain genuinely blocking** and need work before Phase 15 can proceed against them: `search-taxonomy.ts`'s Service-direct-match layer (needs alias-coverage migration first), `get-request-by-id.ts`'s still-rendered `intervention.services` half (needs a replacement display decided), and `seed-taxonomy.ts` (needs a frozen-model equivalent creation script, which does not exist yet — `sync-catalog-to-database.ts` only updates, never creates).
